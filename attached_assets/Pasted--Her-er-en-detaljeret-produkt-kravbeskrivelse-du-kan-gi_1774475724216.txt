## Meeting AI Visualizer — produktbeskrivelse til Replit

### 1. Formål og værdi
**Meeting AI Visualizer** er et værktøj til **live møder og workshops** (primært i industri-/pumpe-kontekst, herunder **Grundfos** som referencekunde). App’en skal:

1. **Transskribere tale til tekst i realtid** (dansk og/eller engelsk, ofte blandet med fagtermer).
2. **Automatisk generere en visuel repræsentation** af det, der tales om, som **HTML med indlejret CSS** — ikke bare generisk tekst, men fx diagrammer, “mock”-HMI, user journey-kort, workflow, produkt/krav-overblik osv., afhængigt af emnet.
3. **Dele tilstand live** mellem flere deltagere (samme “rum”), så alle ser **samme visualisering** og en **fælles transskriptionslinje** (med forbehold for hvordan STT kører pr. klient).

**Kerneprincip:** Deltagerne får et **fælles visuelt anchor** under diskussionen, så abstrakte emner (pumper, krav, flows, grænseflader) bliver konkrete på skærmen.

---

### 2. Målgruppe og typisk brug
- **Brugere:** Tekniske profiler (ingeniører, produkt, projektledelse) i møder om produkt, krav, UX, drift, sikkerhed osv.
- **Sprog:** Primært dansk; app’en skal tåle **dansk/engelsk mix** og **domænesprog** (pumper, HMI, compliance, osv.).
- **Enheder:** Desktop-browser (Chrome/Edge anbefalet til Web Speech); andre kan deltage via link til samme rum.

---

### 3. Hovedfunktioner (hvad app’en skal kunne)

#### 3.1 Transskription (venstre panel)
- **Browser-tilstand:** Web Speech API med dansk (og evt. engelsk) — starter/stopper med optagelse.
- **Meeting-tilstand (valgfri):** **Deepgram** streaming (fx Nova-2/Nova-3) via WebSocket; API-nøgle må **ikke** ligge hardcoded i frontend — server udleverer **kortlivet token** til browseren (`GET /api/deepgram-token`), med mulighed for at slå browser-udlevering fra i produktion (env).
- **Normalisering før brug:** Rå STT skal **ikke** bruges ukritisk. Der skal være et lag der:
  - fjerner/reducerer **fyldord** (da/en),
  - retter **typiske ASR-fejl** i industrikontekst (ordbog/fraser, kontekstuelle regler med “lav risiko”),
  - bruger **én fælles konfig** (fx JSON med lexikon + regler) som både klient og server kan læse.
- **Fleksibel visning:** Lokalt transskriberet tekst + evt. **rum-segmenter** med taler-navn og tidsstempel.

#### 3.2 Visualisering (højre panel)
- Brugeren kan **manuel** trigge visualisering eller få **automatisk** visualisering med interval (fx hvert 45. sekund), **kun når** der er nok ny tekst (minimum ord / nye ord siden sidste viz — med konsistent ordtælling mellem check og API-kald).
- Server kalder **Anthropic (Claude)** med en **systemprompt** der låser:
  - **Grundfos-relateret visuelt sprog** (brandfarver, HMI/SCADA-palet til visse typer),
  - **forskellige visualization families** (fx HMI/dashboard, user journey, workflow/process, fysisk produkt, krav/spec, management-overblik),
  - at output er **HTML + CSS** i et bestemt format.
- **Klassifikation på serveren** (uden ekstra LLM-kald): ud fra **normaliseret** transskription vælges emne/familie, så modellen får **låste instruktioner** (“lav kun denne type viz”). Det skal undgå at alt bliver samme skabelon (fx HMI) når brugeren taler om journey, hardware, osv.
- **Inkrementelle opdateringer:** Når der allerede findes en visualisering, kan næste kørsel sende **tidligere HTML** med og bede modellen **bygge videre** (samme familie / kompatibel gruppe). Ved **emneskift** skal server/klient kunne **droppe** previous HTML og starte forfra.
- **Streaming:** Svar fra Claude streames som **SSE** til browseren, så brugeren ser progressiv opbygning (ikke først alt til sidst).
- **Kvalitetsgate:** Hvis genereret HTML er for kort/incomplet, skal server **ikke** erstatte en god eksisterende viz med skrammel.

#### 3.3 Rendering — kritisk teknisk krav
- Genereret HTML/CSS må **aldrig** indsættes i appens DOM med **`innerHTML` på hovedlayoutet** (det ødelægger appens egne styles).
- **Korrekt:** Vis output i et **dedikeret `<iframe>`** (`contentDocument.write` + `close`).

#### 3.4 Multi-user “rum”
- Opret/join **6-tegns rumkode** (fx i query `?room=XXXXXX`).
- **Server-Sent Events (`GET /api/sse`):** broadcast af
  - nye **transskriptionssegmenter**,
  - ny **visualisering** (HTML + metadata),
  - **deltagerliste**.
- Klient sender **segmenter** til server (`POST /api/segment`), som **merger** og broadcaster.
- **Visualisering:** Én kørsel kan opdatere alle i rummet (hvem der trigger kan afhængige af produktvalg — typisk den der visualiserer).

#### 3.5 Video (valgfri integration)
- **Jitsi Meet** (fx `meet.jit.si`) som **flydende vindue** / iframe — ikke kernen i produktet, men understøttet i UI.

#### 3.6 Historik og debug (prototype-niveau)
- **Mødehistorik** på server (i dag ofte **in-memory** — kan mistes ved restart; persistent lagring er ønskelig fremadrettet).
- **Visualiserings-debug** til udviklere/test: snapshot af request-body, merged transskription, server meta (klassifikation, merge-strategi, incremental-beslutning), evt. **samlet export** til udklipsholder.

#### 3.7 Sikkerhed og drift
- **API-nøgler kun i env** (`.env` / Replit Secrets): `ANTHROPIC_API_KEY`, valgfrit `DEEPGRAM_API_KEY`.
- **Rate limiting** på `/api/visualize` (per IP).
- **Max body size** på visualize-endpoint.
- Fejlresponser må **ikke** lække hemmeligheder eller interne stier.

---

### 4. Backend-endpoints (kontrakt — kan implementeres i Express/Fastify på Replit hvis ønsket)
Originalen bruger **Node `http` uden framework**; på Replit må I gerne bruge et framework, så længe adfærden bevares:

| Metode | Sti | Formål |
|--------|-----|--------|
| GET | `/` | Single-page UI (HTML) |
| GET | statiske filer | `public/` (CSS/JS hvis split) |
| POST | `/api/visualize` | JSON body → stream SSE med chunks + `done` + meta |
| GET | `/api/deepgram-token` | JSON med key/token til browser (hvis feature er på) |
| GET | `/api/transcript-normalize.js` eller tilsvarende | Eksponér normaliseringsdata til browser |
| GET | `/api/history` | Liste over gemte møder (hvis implementeret) |
| GET | `/api/sse?room=…` | Server-Sent Events til rum |
| POST | `/api/segment` | Modtag segment, broadcast til rum |

*(Præcis JSON-form for visualize og SSE-events skal matche den eksisterende klient eller tilpasses i fællesskab.)*

---

### 5. Teknologi (reference — Replit kan vælge ækvivalenter)
- **Runtime:** Node.js 18+ (22+ foretrukket).
- **Frontend:** Vanilla HTML/CSS/JS (ingen obligatorisk React; simpelt = lettere at hoste på Replit).
- **AI:** Anthropic Messages API, **streaming**.
- **STT:** Web Speech API + valgfri Deepgram WebSocket.
- **Hosting:** Miljøvariabler for port (Replit sætter typisk `PORT`).

---

### 6. Ikke-funktionelle krav / kendte begrænsninger
- **Ingen rigtige Grundfos backends** — alt er prototype og genereret indhold fra mødetekst.
- **Transskriptionskvalitet** afhænger af mikrofon, støj og STT-leverandør.
- **Klassifikation** er keyword-baseret + heuristikker; den skal løbende finjusteres (rejse vs HMI vs produkt osv.).
- **index.html** kan være stor; refaktor til moduler anbefales.

---

### 7. Miljøvariabler Replit skal understøtte
```
ANTHROPIC_API_KEY=...     # Påkrævet til visualisering
DEEPGRAM_API_KEY=...      # Valgfri — Meeting/Deepgram STT
PORT=...                  # Replit / platform sætter ofte automatisk
ALLOW_DEEPGRAM_KEY_TO_BROWSER=true|false  # Hvis implementeret
```

5. (Valgfrit) Samme multi-user SSE-adfærd som i originalen.
